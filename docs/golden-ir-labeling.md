# 골든 IR 라벨링 프로토콜 (M0b)

작성: 2026-07-14. 계획 정본(`.omc/plans/sketch2pptx-consensus-plan.md` v6.1) §4 M0b 산출물.
좌표계·필드 정의의 정본은 `ir/schema.md`(등방 정규화 정수, 장축 0~1000)이며, 이 문서는
**그 스키마에 맞는 "정답(골든) IR"을 어떻게 만들고 검증하는가**를 규정한다.

## 0. 왜 이 문서가 중요한가 (R11)

골든 IR은 M0b의 모든 계측 — recall/precision/F1, centroid 오차, IoU, role confusion,
의도-기반 과교정/미교정 카운트(AC2 기계 백업), AC7 diff 추정 — 의 **공동 담체**다.
골든이 틀리면 게이트 전체가 조용히 오염된다(계획 리스크 R11).

이것은 이론적 우려가 아니라 **실증된 사실**이다. M0a에서 1차 계측의 ex02 p90이
39.8‰로 튀었는데, 추적 결과 인식이 아니라 **골든 라벨 자체의 오류 4+1건**이 원인이었다
(`_thinking/analysis/001-m0a-probe.md` §5). 대표 사례:

| 항목 | 골든 v1.0 (오류) | 진실 (픽셀 실측) | 발견 경로 |
|---|---|---|---|
| ex02 g3 (내부 프레임) | w=586 | w=691 | 인식 2벌 상호 일치 + **골든의 자식 포함관계 자기모순** |
| ex02 n12 (출력 수식) | x=843, w=117 | x=869, w=81 | "⇒" 기호를 bbox에 오포함 |
| ex02 e8 (루프백) | n10→n8 | n11→n8 | 인식 2벌 일치 — "엣지 누락 2건"으로 보였던 것의 실체 |
| ex01 n15 | [893,155] | [882,153] | 인식 2벌 일치, 단 y는 골든이 맞았음(제3값 존재) |

교훈 세 가지가 이 프로토콜의 뼈대다: ① 골든 라벨도 확률적이다 → 단일 라벨링을 믿지 않는다.
② **컨테이너⊇자식 포함관계 기계 검증**은 라벨 자기모순을 공짜로 잡는다 → 필수 절차(§5).
③ **복수 독립 소스 교차중재**가 단순 편차 측정보다 강력한 라벨 QC다 → 표준 절차(§2).

## 1. 대상 파일과 좌표 단위

- 산출물: `tests/fixtures/example-0N.golden.ir.json` — 최소 골든(`*.min-golden.ir.json`,
  bbox+존재+role)에 **eval 전용 확장 필드 `intent_row`·`intent_col`·`size_class`**(§4)를
  더한 전체 골든.
- 좌표는 `ir/schema.md`의 등방 정규화 **정수**: 장축 0~1000, 단축 0~round(1000×단축/장축).
  픽셀→‰ 변환은 두 축 모두 동일 배율을 쓴다:
  `단위길이 = 장축픽셀/1000`, `값‰ = round(픽셀값 / 단위길이)`
  (예: ex01 2366×1402 → 1‰ = 2.366px, y 상한 = round(1000×1402/2366) = 593).
- 오차·허용치 표기는 전부 ‰(장축의 1/1000). 등방이므로 x·y 오차가 동일 물리 척도다.

## 2. 누가, 어떤 도구로 라벨하는가 — 3단 절차 (필수)

라벨링은 단일 행위가 아니라 **작성 → 교차중재 → 실측 판정**의 3단 파이프라인이다.
M0a §5에서 이 절차 그대로 골든 오류 5건을 전부 잡아냈다.

**1단 — 세션 정독 라벨링 (초안 작성).**
메인 세션(Claude)이 손그림 이미지(`_thinking/raws/ref_hand/`)를 Read 도구로 정독하며
스키마에 맞는 IR 초안을 직접 작성한다. 이때 §3 분류 규약과 §6 bbox 경계 규약을 따른다.

**2단 — 독립 소스 교차중재 (분쟁 후보 추출).**
골든을 본 적 없는 **독립 컨텍스트 서브에이전트 ≥2벌**이 같은 손그림을 인식해 IR을 낸다
(M0a의 인식 4벌을 재사용해도 된다). 세 소스(골든 초안 + 인식 N벌)를 대조해:
- **인식 벌들이 서로 일치하는데 골든과만 상충**하는 항목 → 분쟁 후보 (골든 오류 의심).
- 인식 벌들끼리도 갈리는 항목 → 분쟁 후보 (진실 불명, 실측 필요).
- 셋이 다 일치하는 항목 → 통과.

**3단 — 픽셀 실측 판정 (최종심).**
분쟁 후보는 **골든 작성자가 아닌 독립 검증 에이전트**가 원본 이미지에서 색상 마스크·크롭
기반 픽셀 실측(예: PIL로 해당 영역 잘라 잉크 경계 픽셀 좌표를 직접 읽음)으로 판정하고,
§1 공식으로 ‰ 변환해 골든을 정정한다. **자기승인 금지** — 골든을 쓴 컨텍스트가 자기
라벨을 "맞다"고 판정할 수 없다. M0a에서 이 판정이 v1.0→v1.1 정정 5건을 확정했다.

**의도 정보의 최종 소스 = 원작자(사용자).**
bbox는 픽셀 실측이 최종심이지만, `role`·`intent_*`·`size_class`는 "그리는 사람이 무엇을
의도했는가"의 문제라 픽셀에 답이 없다. 시각 단서로 판정이 안 서면 원작자에게 확인하고,
확인 전까지는 보수 기준(§4·§7)을 적용한다.

## 3. 무엇을 노드/컨테이너/엣지로 세는가 (분류 규약)

M0a에서 인식측 "잉여"의 실체가 오류가 아니라 규약 차이였다(캡션을 노드로 셈).
골든과 인식 프롬프트가 **같은 규약**을 써야 diff가 의미를 갖는다:

- **노드** = 화살표(엣지)가 연결되는 **흐름 참여자**. 테두리 유무 무관 — 테두리 없는
  손글씨 스텝도 흐름에 참여하면 노드다.
- **placeholder** = 수식·아이콘·삽입 figure처럼 자동 변환하지 않고 자리만 잡는 원소
  (`kind:"placeholder"` + `placeholder_kind`).
- **annotation** = 순수 캡션·범례·메모(흐름에 참여하지 않는 설명 텍스트). 노드로 세지 않는다.
- **컨테이너** = 다른 원소를 묶는 큰 배경 박스. `children`에는 **직속** 자식만 넣는다
  (손자는 중간 컨테이너의 children으로).
- **엣지** = 화살표 잉크의 실제 양끝이 닿는(가장 가까운) 도형 쌍을 `from`/`to`로.
  머리 방향으로 `arrow`(single/double/none), 관찰 형태로 `style`(straight/elbow)을 적는다.

## 4. eval 전용 확장 필드 부여 기준 (intent_row / intent_col / size_class)

정의(`ir/schema.md` "M0b 확장 필드" 절과 동일, nodes·containers 공통, eval 전용):

- `intent_row`: string|null — 같은 값 = 의도적으로 같은 가로줄(y중심 정렬 의도)
- `intent_col`: string|null — 같은 값 = 의도적으로 같은 세로줄(x중심 정렬 의도)
- `size_class`: string|null — 같은 값 = 의도적으로 동일 크기(w,h)

부여 기준 — **"손그림의 현재 좌표"가 아니라 "원작자의 의도"를 라벨한다**:

1. **의도 기준**: "이 다이어그램이 PPT로 정서(淨書)됐을 때, 이 박스들이 정렬(동일 크기)
   되어 있지 않으면 원작자가 어색하다고 느낄 그룹"에만 라벨을 준다. 손그림 좌표가 삐뚤어도
   의도가 명백하면 같은 그룹이고, 우연히 좌표가 비슷해도 의도가 없으면 다른 그룹이다.
2. **판정 근거 우선순위**: ① 원작자 확인 > ② 도식 의미(같은 파이프라인 단계·병렬 분기·
   동종 반복 스텝) > ③ 기하적 줄 맞춤 시도의 흔적. ③ 단독으로는 라벨하지 않는다.
3. **애매하면 null (보수 원칙)**: 이 라벨은 `count_corrections`의 미교정(intent 그룹인데
   정렬 안 됨)·과교정(intent 없는데 크게 이동) 판정 기준이다. 과잉 라벨은 정규화에 억울한
   미교정을, 과소 라벨은 억울한 과교정을 만든다 — 확신 없는 쪽은 null로 두고 판정을
   move_tol에 맡기는 것이 덜 위험하다.
4. **표기 규약**: 그룹 이름은 파일 내 유일 문자열, 관례상 행 `r1,r2,…`, 열 `c1,c2,…`,
   크기 `s1,s2,…`. 한 박스가 `intent_row`와 `intent_col`을 동시에 가질 수 있다.
   그룹 크기는 2 이상이어야 의미가 있다(원소 1개짜리 그룹 금지 — 기계 검증이 잡는다).
5. **size_class 주의**: 텍스트 길이 때문에 폭이 **의도적으로** 다른 박스는 같은 class가
   아니다. "동일 계열 스텝이라 같은 크기로 그리려 했다"가 확인될 때만 묶는다.

## 5. bbox 경계 규약과 좌표 정밀도 허용치

**경계 규약** (M0a 오류 2번 — "⇒" 오포함 — 의 재발 방지):
- bbox = **그 도형 자체의 잉크 외접 사각형**. 테두리가 있으면 테두리 스트로크 외곽 기준,
  테두리 없는 손글씨 노드면 글자 잉크의 외접 사각형.
- 인접한 화살표·연결 기호(⇒ 등)·캡션·번호 매김은 **제외**한다. 그것들은 각각
  엣지/annotation의 소관이다.

**정밀도 허용치**:
- 좌표는 ‰ 정수로 기록한다(반올림 오차 ±0.5‰는 불가피, 무시).
- 개별 좌표의 목표 정밀도: 픽셀 실측 대비 **±3‰ 이내**.
- 라벨 품질 게이트(§7 스팟체크): **centroid 재라벨 편차 중앙값 ≤ 좌표 서브게이트 X의 절반**.
  X는 M0b 2c(합성 노이즈 스윕)에서 확정된다(도출 순서: X < align_tol < 붕괴 오차).
  **확정 전 잠정값 X=6‰**(정규화 기본 tol 하한과 동일)를 쓰면 허용치는 **3‰** — 이는
  M0a에서 관측된 골든↔독립 인식 편차 중앙값 2.2~3.0‰(라벨 오차+인식 오차 합의 상한)과
  정합적이다. 2c 확정 후 이 수치만 갱신한다(절차는 불변).

## 6. 기계 검증 (필수 — 골든 확정 전 반드시 실행)

골든을 버전 확정하기 전에 아래 검증을 **스크립트로** 돌리고 결과를 기록한다.
M0a에서 g3 오류(w=586→691)를 잡아낸 것이 바로 포함관계 자기모순 검사였다 — 이 검증은
추가 라벨 비용 0으로 라벨 오류를 잡는 가장 싼 안전망이다.

1. **스키마 검증**: `ir/schema.json`에 대해 구조 검증(필수 필드·타입·허용값).
2. **좌표 범위**: 모든 bbox가 `0 ≤ x`, `x+w ≤ 1000`(장축), `0 ≤ y`, `y+h ≤ 단축 상한`,
   `w,h > 0`.
3. **컨테이너⊇자식 포함관계 (핵심)**: 모든 컨테이너 c와 `c.children`의 모든 자식에 대해
   자식 bbox가 c의 bbox 내부에 포함되어야 한다. 손그림 선 두께·반올림을 감안해
   **변당 돌출 ≤ 3‰까지만 허용**, 초과 시 라벨 오류로 판정하고 픽셀 실측으로 재중재.
4. **참조 무결성**: `children`·`from`/`to`의 모든 id가 실존, 전체 id 유일, 각 원소의
   부모 컨테이너는 최대 1개, 컨테이너 중첩에 사이클 없음(트리).
5. **기하-소속 교차 경고**: 어떤 컨테이너 bbox 내부에 기하적으로 완전히 들어있는데
   그 컨테이너(또는 그 하위)의 children에 없는 박스 → 경고. 의도적 비소속인지 확인한다
   (M0a g3 오류의 역방향 탐지기).
6. **intent 그룹 sanity**: 원소 1개짜리 intent/size 그룹 → 오류. 같은 `intent_row` 그룹의
   y중심 산포, `size_class` 그룹의 w·h 산포를 **정보용으로 출력**(손그림이라 클 수 있음 —
   게이트가 아니라 라벨 실수 의심용 참고치).

## 7. 재현성 스팟체크 (필수 절차)

개인 프로젝트라 '2인 교차 라벨'이 불가능하므로, **시차 재라벨**로 라벨러 내 일관성을
수치화한다. M0a의 정식 스팟체크는 무산출로 중단되어 **M0b step 0으로 이월**된 상태다
(`001-m0a-probe.md` §7) — 즉 이번 라벨링에서 반드시 수행한다.

절차:
1. 전체 골든 완성 후, **이전 라벨 파일을 열지 않은 상태**(독립 컨텍스트 서브에이전트,
   또는 충분한 시차를 둔 세션)에서 동일 손그림의 대표 부분집합(박스 최소 10개 이상,
   컨테이너·placeholder 포함)을 재라벨한다.
2. 원 골든과 대조해 측정: ① bbox centroid 편차 중앙값(‰), ② role 불일치율,
   ③ intent 라벨 불일치율(intent_row·intent_col·size_class — 그룹 분할이 동형인지,
   즉 "같은 그룹으로 묶였는가"를 쌍 단위로 비교).
3. **허용치**: centroid 재라벨 편차 중앙값 ≤ **X의 절반**(잠정 3‰, §5), role·intent
   불일치 ≤ **5%**.
4. 초과 시: 불일치 항목을 §2의 3단 절차(교차중재→픽셀 실측/원작자 확인)로 판정해
   프로토콜의 모호한 조항을 조인 뒤 **재라벨하고 스팟체크를 반복**한다.
5. 결과(편차 수치·불일치 목록·조치)를 `_thinking/analysis/` 분석 문서에 기록한다.

## 8. AC2 판정자 캘리브레이션 (M0b 존재적 게이트의 전제)

M0b의 존재적 게이트는 "PowerPoint에서 위치 손본 박스 수 / 전체"를 마진 밴드
(≤7% GO / 7~13% 표본 추가 / >13% NO-GO)로 판정하는데, 판정자(사용자)의 카운트 자체가
흔들리면 밴드가 무의미하다. 정렬 지각은 색 그룹핑·테두리 등 스타일에 부분 의존하므로
**무스타일 스텁 위 판정의 신뢰도**를 먼저 계측한다:

1. 같은 스텁 pptx 1벌을 **시차를 두고 2회** 판정한다(1회차 카운트를 다시 보지 않은 상태).
2. 카운트 규약: **설문 (1) "위치·정렬·크기가 틀려서 손봐야 하는 박스"만** 센다.
   (2) 스타일 조악, (3) 커넥터 경로 조악은 세지 않는다(커넥터는 elbow 게이트가 책임).
3. 2회 카운트의 편차(%p)를 기록한다. **편차 > 밴드 폭 ±3%p**이면: 무엇을 "위치 틀림"으로
   셀지 판정 기준을 명문화(예: "인접 박스와 시각적으로 6‰ 이상 어긋나 보이면 카운트")한
   뒤 재판정한다.
4. 캘리브레이션 결과는 존재적 게이트 판정 기록과 함께 `_thinking/analysis/`에 남긴다.
