## 2. 딥인터뷰에서 확정된 것 (5라운드, 각 라운드 = 질문 1개)

| # | 질문 | 사용자 답 → 확정 사항 |
|---|---|---|
| R1 | 검수 방식? | **바로 pptx 생성 → PowerPoint에서 직접 검수** → 채팅 지적 → IR 수정 → 재렌더(증분 파일명) 또는 PPT 손수정. 오버레이/HTML 중간물은 만들지 않음 (001의 원안 폐기) |
| R2 | 완성 PPT 원본 .pptx 보유? | **있음. 스타일 2종 — 글꼴만 다르고 색·구조 동일** → 스타일 config = 원본 추출 + 글꼴 스위치 |
| R3 | 다이어그램 유형 범위? | **박스-화살표가 대다수.** 수식·아이콘은 가끔 → 자동 변환 않고 **플레이스홀더 박스**로 위치만 표시 (수동 삽입) |
| R4 | (컨트래리언) 무엇이면 도구를 버리나? | **"재배치 지옥"** → 성공 기준 1순위 = 생성 직후 재배치 필요 박스 ≤10%. 속도는 부차 기준 |
| R5 | 실행 형태? | **Claude Code 세션** (인식기 = 세션의 VLM, 렌더러 = 세션이 호출하는 파이썬 스크립트) |

R4가 가장 중요한 수확이다 — 이 프로젝트의 품질 핵심이 인식 정확도가 아니라
**레이아웃(정렬·간격·크기) 품질**임이 여기서 확정됐고, 이후 합의 5라운드 전체가
사실상 "재배치 ≤10%를 어떻게 유효하게 측정·보장하는가"를 두고 벌어졌다.

## 3. 합의 루프 이력 — 각 라운드가 무엇을 잡았나 (사고의 흐름)

Planner가 초안(v1)을 쓰고, 매 라운드 Architect(아키텍처 검토, 최강 반론 의무)와
Critic(품질 기준 8개 평가)이 순차 검토 → Planner가 반영. 5라운드 내내 REVISE였지만
반려 사유가 점점 좁아지는 **수렴 궤적**이었다:

- **라운드 1 — "순환 프록시" 적발:** v1의 M0 게이트("정규화 후 정렬율 ≥90%")는
  알고리즘이 스냅하기로 한 걸 스냅했는지만 재는 자기참조 지표였다. 과교정(의도된 오프셋까지
  뭉갬)이 심할수록 점수가 **오히려 오르는 역상관** — 재배치 지옥을 전혀 못 잡는다.
  → M0를 "얇은 수직 슬라이스"로 확장해 실물 pptx에서 사람이 손본 박스 수를 직접 센다.
- **라운드 2 — 측정 유효성 3중 결함:** ① 렌더 대상이 골든(정답) IR인지 세션 인식 IR인지
  미명시 → 인식 오차가 정규화 탓으로 오귀속 ② 좌표 기하(bbox 위치 정확도)를 아예 안 잼
  ③ 대체 지표도 임계값 상대 정의라 정상 구현에서 자동 0. → 골든/인식 경로 분리, 좌표
  서브게이트(centroid/IoU) 신설, 의도(intent_align) 기반 지표로 교체.
- **라운드 3 — "반사실 세계" 범주 오류:** v3가 1순위 게이트를 골든 경로로만 판정했는데,
  사용자는 언제나 세션 인식 경로에 산다. 골든 경로 통과 ≠ 사용자 경험. → pass/fail을
  **사용자가 실제 겪는 세션 인식 경로**로 승격, 골든/좌표/role 분해는 "fail 원인이 정규화냐
  인식이냐"를 가려주는 **귀속(attribution) 레이어**로 재배치.
- **라운드 4 — 채택안 자체의 미검증:** 정규화 전략 B1(클러스터링)의 우월성이 원리론뿐이었다
  — 좌표가 정확하면 B2(단순 스냅)로 충분(B1은 과잉설계), 부정확하면 B1도 붕괴. → **M0에서
  B1 vs B2를 실제 인식 좌표 위에서 A/B 실증**(B2가 대등하면 YAGNI로 B2 채택). IR 좌표계
  0~1000 정규화 확정, 밀집 elbow 커넥터 라우팅을 명시 게이트(R13)로 승격.
- **라운드 5 — 리스크-우선순위 규율 복원:** ① M0가 너무 비대해져 "값싼 조기 손절"이라는
  존재 근거를 침식 → **M0a(반나절 벌거벗은 인식-좌표 프로브, 손절 게이트) / M0b(통과 시에만
  본 건설) 분리** ② R13 복구책이 "PowerPoint가 저장된 커넥터 경로를 존중한다"는 미검증
  전제 위 → 실측 스파이크 통과 후에만 채택으로 강등 ③ 좌표계 이방성(16:9에서 1‰(x)≠1‰(y))
  → **등방**(장축 0~1000, 단축 비례)으로 확정.
- **사후 검증 패스 (v6에 대해, 별도 critic 1회):** 5라운드 상한 종료 시점의 v6가 라운드5
  피드백을 실제 본문에 반영했는지만 검증 → **APPROVED.** 신규 모순 없음, 잔여 cosmetic
  2건(총소요 산술, ADR 소요 표기)은 v6.1에서 정정.

교훈으로 남길 것: 다섯 라운드 모두 "코어 설계(IR 단일 계약 + 결정론 렌더러)는 건전"이라는
판단은 흔들리지 않았다. 반려는 전부 **측정과 게이트의 유효성** 문제였다 — "재배치 ≤10%"라는
한 문장을 실제로 측정 가능하게 만드는 데 다섯 라운드가 들었다.

## 4. 최종 계획 핵심 (상세는 계획 본체 참조)

**아키텍처 (확정):** 손그림 이미지 → [세션 VLM 인식] → IR(JSON, 등방 0~1000 좌표) →
[레이아웃 정규화: B1 클러스터링 vs B2 스냅, M0에서 A/B로 결판] → [python-pptx 결정론
렌더러 + 스타일 config(원본 pptx 추출, 글꼴 2종 스위치)] → 편집 가능 pptx (증분 파일명)
→ PowerPoint 직접 검수 → 채팅 수정 루프.

**마일스톤 (총 7.5~9 작업일):**

| 단계 | 내용 | 소요 | 핵심 게이트 |
|---|---|---|---|
| **M0a** | 벌거벗은 인식-좌표 프로브 (렌더러 없이 세션 bbox vs 최소 골든) | 반나절 | 좌표 충실도 가망 없으면 **여기서 손절** |
| **M0b** | 인식 계측 완성 + 스텁 렌더 + B1/B2 A/B + elbow 게이트 | 2~2.5일 | **AC2 재배치율: 전 표본 ≤7% GO / 하나라도 >13% NO-GO / 그 외 표본 추가** (최악치 기준) |
| M1 | 완전 충실도 렌더러 (스타일·커넥터 바인딩·한글 ea 폰트) | 2일 | 스타일 AC5 3종 검사, 그룹/커넥터 스파이크 |
| M2 | 검수 루프 (IR 수정→재렌더, align_hint 채팅 교정) | 1일 | 전체 소요 ≤10분 (AC7) |
| M3 | 품질 고도화 + 새 손그림 2~3장 실전 재검증 | 2~3일 | AC2 재확인 |

**리스크 1순위(R1) = 재배치 지옥, 최상류 병목(R10) = VLM 좌표 정확도** — M0a가 R10을
반나절에 먼저 게이트하고, M0b가 R1을 실물로 게이트하는 구조다.

## 7. 추가 기록 (2026-07-14 append) — 지속 참고 PPT 유입 워크플로

사용자 질문: "앞으로 지속적으로 완성 PPT가 나올 텐데, 그때마다 참고용으로 계속 전달할 수 있게 되어 있나?"

**판정: 구조적으로는 지원, 명시 절차는 미정의.**
- 지원되는 이유: 스타일이 코드가 아니라 **config 파일(원본 .pptx에서 추출) + 결정론 렌더러**로
  분리돼 있어서(계획의 핵심 결정), 새 완성 PPT가 생기면 추출기를 다시 돌려 config를 갱신하는
  것만으로 반영된다. 아키텍처 변경 불필요.
- 미정의인 부분: 새 PPT를 어디에 두는지, 언제 재추출하는지, 기존 config와 충돌 시 처리 —
  현재 계획에는 M4 follow-up "config 자동추출 정밀화" 한 줄뿐.

**합의된 추가 방향 (저비용 3건):**
1. **`references/` 폴더 규약** — 새 완성 PPT 수집 지점 (던져 넣기만 하면 됨)
2. **재추출 = 덮어쓰기 금지, diff 보고 후 사용자 승인 갱신** — 새 추출값이 기존 config와
   다르면 "테두리색 변경 감지, 갱신?"을 보여주고 사용자가 결정. 이유: 매번 자동 반영하면
   config가 흔들려 "같은 IR인데 지난주와 다른 렌더 결과"가 나오는 스타일 드리프트 발생
   (재현성 = Principle의 결정론 렌더러 취지 위배). 사용자 전역 규칙(산출물 no-overwrite)과도 정합.
3. **새 완성 PPT의 이중 용도** — 스타일 참고 + 골든 픽스처(렌더러 회귀 테스트 재료)로 재활용.

**상태:** 계획 본체(.omc/plans)에는 아직 미반영 — 실행 승인 시 M4 follow-up(또는 M1 extract.py
설계 시점)에 위 3건을 편입한다. 본 append가 그 근거 기록.

**정정 (같은 날):** §6 말미의 "아직 git repo 아님"은 이 append 직전에 해소됨 — repo 생성·push 완료
(`git@github.com:dlacksdn/ppt_creator.git`, main). commit/push는 전역 정책대로 분기마다 자율 수행.
