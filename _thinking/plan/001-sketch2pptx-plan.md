# 001 — 손그림 다이어그램 → 내 스타일 PPT 자동 변환 (계획 v1)

작성: 2026-07-14. 이 문서는 ppt_creator 프로젝트의 첫 계획 문서다.
아직 코드가 한 줄도 없는 상태이며, 이 문서 하나만 읽으면 다른 세션이 바로 구현을 시작할 수 있게 쓴다.

---

## 1. 배경 — 이 프로젝트가 왜 필요한가

사용자는 발표 자료(PPT)에 다이어그램을 넣기 전에 항상 **태블릿에 손으로 먼저 그린다**.
그 다음 PowerPoint에서 손그림을 보면서 도형을 하나하나 다시 그리는데, 이 "옮겨 그리기"가
시간이 오래 걸리고 번거롭다 (예: 박스 40개짜리 아키텍처 도식이면 30분~1시간).

**목표**: 손그림 사진 + 내 PPT 스타일 → **편집 가능한** .pptx 파일을 자동으로 만들어주는 도구.

- 출력은 반드시 PowerPoint에서 직접 수정 가능한 **네이티브 도형**이어야 한다 (이미지 삽입 ❌).
- 스타일은 사용자가 기존에 만들어 온 PPT 양식(색, 폰트, 테두리, 정렬 방식)을 따라야 한다.
- 실제 예시 쌍이 이미 있다: LVDM 아키텍처 손그림 ↔ 같은 내용의 완성 PPT 슬라이드.

**성공 기준(MVP)**: 예시 손그림 수준의 다이어그램을 넣었을 때,
손으로 처음부터 다시 그리는 것보다 확실히 빠르게 (검수 포함 10분 이내에)
내 스타일의 편집 가능한 pptx가 나온다. 100% 자동 완벽이 목표가 아니다 —
**"90% 자동 + 10% 검수"가 목표다.**

---

## 2. 핵심 설계 아이디어 (가장 중요한 부분)

이 문제를 "이미지 → 이미지" 변환으로 풀면 안 된다.
**"이미지 → 구조화 데이터 → 렌더링"** 3단계로 쪼갠다.

```
손그림 이미지 ──(A) 인식──▶ IR (JSON) ──(B) 검수/수정──▶ 확정 IR
                                                          │
스타일 config ────────────────────────────(C+D) 정규화+렌더링──▶ .pptx
```

용어 하나만 알면 된다: **IR (Intermediate Representation, 중간 표현)**
= 다이어그램의 내용을 담은 JSON. "어떤 박스가 어디에 있고, 무슨 텍스트가 적혀 있고,
어떤 화살표가 무엇을 연결하는가"만 담는다. 색·폰트 같은 꾸밈은 담지 않는다.

**왜 이렇게 쪼개나 (핵심 근거):**

| 역할 | 담당 | 성질 |
|---|---|---|
| 손그림 "읽기" (인식) | VLM (Claude 비전) | 확률적 — 틀릴 수 있음 |
| 스타일 입혀 "그리기" (렌더링) | 결정론적 파이썬 코드 | 항상 같은 결과 |

- LLM에게 pptx를 직접 그리게 하면 **매번 결과가 다르고 스타일이 흔들린다.**
- 반대로 인식을 규칙 기반(OpenCV 등)으로 하면 손글씨·삐뚤한 박스를 못 읽는다.
- 그래서 **틀릴 수 있는 부분(인식)은 LLM에, 항상 같아야 하는 부분(스타일)은 코드에** 맡긴다.
- IR이 중간에 있으므로 "인식이 틀렸을 때 IR만 고치면 되고", "스타일을 바꾸고 싶으면
  config만 바꿔 재렌더링하면 된다". 수정 요구사항이 자연스럽게 해결된다.

## 3. 단계별 상세 설계

### (A) 인식: 손그림 → IR JSON

- 입력: 손그림 이미지 1장 (+ 도메인 용어 사전, 예: "LoRA", "Cross-Attention" — 손글씨 오독 방지)
- 방법: VLM에게 이미지를 주고 아래 스키마의 JSON을 뽑게 한다.
  MVP에서는 **Claude Code 세션 자체가 인식기** 역할을 한다 (이미지 Read → JSON 작성).
  API 자동화는 나중 문제.
- IR 스키마 초안 (v0.1 — M1에서 확정):

```json
{
  "canvas": {"w": 1280, "h": 720},
  "nodes": [
    {"id": "n1", "kind": "box",       "bbox": [x, y, w, h], "text": "Base Model",
     "role": "default"},
    {"id": "g1", "kind": "container", "bbox": [x, y, w, h], "title": "Action Control Module",
     "children": ["n1", "n2"]}
  ],
  "edges": [
    {"id": "e1", "from": "n1", "to": "n2", "arrow": "single", "label": "Down",
     "style": "elbow"}
  ],
  "annotations": [
    {"kind": "legend", "bbox": [x, y, w, h], "items": ["Trainable", "Frozen"]}
  ]
}
```

- 좌표는 **손그림에서 읽은 대략적 위치를 그대로** 담는다 (정규화는 D 단계 담당).
- `role`은 스타일 매핑 열쇠다: `default`(흰 박스) / `emphasis`(분홍 강조) /
  `input`(하늘색) / `output`(초록) 등 — 사용자 스타일에서 관찰되는 의미 단위.

### (B) 검수: 사람이 IR을 확인·수정 (이 단계가 UX의 핵심)

- 인식 정확도는 예시 수준 밀도(박스 40개, 중첩 컨테이너)에서 80~95%로 예상.
  화살표 방향 오류·라벨 누락이 반드시 몇 개 나온다. **원샷 완벽을 기대하는 설계는 실패한다.**
- MVP 방식: IR을 **HTML/SVG로 간단히 시각화**해서 원본 손그림과 나란히 보여준다
  → 사용자가 "e3 화살표 방향 반대", "n7 텍스트 오타" 식으로 지적 → IR 수정 → 재렌더.
- 검수 1~2분 ≪ 손으로 다시 그리기 30분. 여기서 시간이 절약된다.

### (C) 스타일: config 파일 1개

- 사용자 스타일을 YAML/JSON config로 **한 번만** 정의한다. 예:

```yaml
font: {name: "Arial", size_pt: 10, title_size_pt: 14, bold_titles: true}
roles:
  default:   {fill: "FFFFFF", line: "1F3864", line_w_pt: 1.25}
  emphasis:  {fill: "F8D7DA", line: "1F3864"}   # 분홍 강조 박스
  container: {fill: "F2F2F2", line: "000000", title_pos: "top-center"}
edges:
  default: {type: "elbow", color: "1F3864", width_pt: 1.5, arrow: "triangle"}
layout: {grid_px: 8, row_align_tol: 20, uniform_box_h: true}
```

- 최초 config는 사용자의 기존 완성 PPT 슬라이드를 보고 **손으로 뽑는다** (반나절 작업).
  pptx에서 자동 추출(XML 파싱)은 M3의 개선 항목이지 MVP 필수가 아니다.
- 중요한 통찰: 스타일 = 색·폰트만이 아니라 **정렬·간격·크기 통일 규칙**까지다.
  예시 두 이미지의 진짜 차이는 "줄 맞춤과 박스 크기 통일"이다.

### (D) 레이아웃 정규화: 스케치 배치를 존중하며 다듬기

- 원칙: **배치는 사용자가 태블릿에서 이미 끝냈다. 새로 배치하지 않는다.**
  (graphviz 같은 자동 레이아웃 ❌ — 사용자 의도 훼손)
- 하는 일 (전부 결정론적 규칙):
  1. 그리드 스냅 (좌표를 8px 격자에 맞춤)
  2. 행/열 정렬 (y좌표 차이가 임계값 이내인 박스들 → 같은 줄로 정렬)
  3. 크기 통일 (비슷한 크기의 박스 → 동일 크기로)
  4. 컨테이너 패딩 통일 (자식 박스와 테두리 간격 일정하게)
- 함정: 과교정 주의. 임계값을 보수적으로 잡아 "일부러 다르게 그린 것"을 뭉개지 않는다.

### (E) 렌더링: IR + config → .pptx

- 도구: **python-pptx** (pptx는 열린 XML 포맷 OOXML이라 완전 제어 가능)
- 생성물: 네이티브 도형(사각형·텍스트박스·커넥터·그룹) → PowerPoint에서 자유 수정 가능
- 커넥터는 도형에 **바인딩**한다 (박스를 옮기면 화살표가 따라오게).
  python-pptx 기본 API가 약한 부분(elbow 커넥터, 화살표 머리)은 XML 직접 조작으로 해결
  — 알려진 패턴이 있는 영역이다. M1에서는 단순 직선 커넥터로 시작해도 된다.
- 렌더러는 순수 함수처럼: 같은 IR + 같은 config → 항상 같은 pptx.

## 4. 마일스톤 (작게 시작해서 게이트마다 확인)

| 단계 | 내용 | 산출물 | 게이트 (통과 기준) |
|---|---|---|---|
| **M0 스파이크** | 예시 손그림 1장을 VLM으로 IR 추출해보고 정확도 실측 | IR JSON 1개 + 오류 목록 | 노드/엣지 80%↑ 인식되면 GO |
| **M1 MVP** | IR 스키마 확정 + 스타일 config 수동 작성 + python-pptx 렌더러 | 첫 .pptx (예시 재현) | PowerPoint에서 열리고 편집 가능 |
| **M2 검수 루프** | IR→SVG/HTML 미리보기 + 수정→재렌더 워크플로 | 검수 도구 | 전체 소요 10분 이내 |
| **M3 품질** | elbow 커넥터/화살표 XML, 레이아웃 정규화 고도화, config 자동 추출 | 개선판 | 새 손그림 1장으로 실전 검증 |
| **M4 (선택)** | Claude Code 스킬화 or 서비스화 | /sketch2pptx 스킬 | — |

M0가 가장 중요하다: **인식 정확도가 이 프로젝트의 유일한 진짜 리스크**이므로,
렌더러를 만들기 전에 반나절짜리 스파이크로 먼저 확인한다 (안 되면 여기서 멈추면 손해 없음).

## 5. 리스크와 대응

| 리스크 | 가능성 | 대응 |
|---|---|---|
| 밀도 높은 그림에서 인식 누락/오류 | 높음 (확실히 발생) | (B) 검수 단계를 워크플로에 내장, 용어 사전 제공 |
| python-pptx 커넥터 표현력 한계 | 중간 | XML 직접 조작 (M1은 직선으로 우회) |
| 레이아웃 정규화가 원본 의도 훼손 | 중간 | 임계값 보수적, 정규화 on/off 플래그 |
| 손글씨 오독 (흘림체) | 중간 | 도메인 용어 사전 + 검수에서 잡힘 |
| "내 스타일" 정의가 생각보다 애매 | 중간 | config 초안을 예시 PPT 기반으로 만들고 반복 수정 |

## 6. 기각한 대안 (왜 안 했나)

- **LLM이 python-pptx 코드를 직접 생성**: 매번 결과가 달라 스타일 일관성 없음.
  단 M0 스파이크 용도로는 허용 (빠른 검증).
- **자동 레이아웃 (graphviz류)**: 태블릿에서 이미 완성한 배치를 파괴함.
- **SVG → PPT 변환 경유**: 변환 결과가 프리폼 덩어리라 편집성이 나쁨.
- **이미지로 출력**: 요구사항 자체 위배 (수정 불가).
- **OpenCV 규칙 기반 인식**: 손글씨·삐뚤한 도형에 취약, VLM 대비 이득 없음.

## 7. 기술 스택 (예정)

- Python 3.11+ 전용 venv (`ppt_creator/.venv`)
- `python-pptx` (렌더링), `lxml` (커넥터 XML 조작), `pyyaml` (config)
- 인식: MVP는 Claude Code 세션 직접 → 이후 Claude API (vision) 자동화
- 검수 미리보기: 단순 HTML/SVG 생성 (의존성 없음)
