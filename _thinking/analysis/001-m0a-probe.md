# 001 — M0a 벌거벗은 인식-좌표 프로브 결과 (손절 게이트 판정)

작성: 2026-07-14. 계획 정본(`.omc/plans/sketch2pptx-consensus-plan.md` v6.1) §4 M0a / §7 0-pre~2a의 실행 기록.
**판정 먼저: GO — 손절 사유 없음.** 세션 VLM의 bbox 좌표 충실도는 계획이 우려한 수준을 크게 상회한다.

## 1. 이 문서의 위치

M0a는 이 프로젝트의 최상류 하중 전제 — **"세션 VLM이 쓸 만한 bbox 좌표를 산출하는가"(R10)** — 를
렌더러·정규화·A/B를 짓기 전에 반나절 비용으로 단독 검증하는 손절 게이트다 (합의 라운드5의 산물).
여기서 가망이 없으면 프로젝트 전체를 반나절 손실로 접는 구조였다. 결과는 정반대로 나왔다.

## 2. 재료

- 손그림 13장 전수 조사 (`_thinking/raws/ref_hand/`, 서브에이전트) → 골든 표본 2장 선정:
  - **example-01** = `KakaoTalk_20260714_142818087.jpg` (2366×1402): 박스 40개(노드 30+컨테이너 10),
    중첩 깊이 3~4, 범례 포함 — 최고밀도 스트레스 케이스
  - **example-02** = `KakaoTalk_20260714_143436894.jpg` (1885×1376): 중밀도 15개, **한글 비중 최고**,
    수식·논문 figure 삽입(placeholder 검증), 테두리 없는 손글씨 스텝 포함 — 이질 케이스
- 좌표계: 등방 정규화 정수 (장축 0~1000; ex01 y≤593, ex02 y≤730) — `ir/schema.md` 0-pre에서 확정

## 3. 방법 (오염 방지 설계)

- **골든**: 메인 세션이 정독 라벨링 (`tests/fixtures/example-0N.min-golden.ir.json`)
- **인식**: 골든을 본 적 없는 **독립 컨텍스트 서브에이전트 4벌** — 표본 2 × (무힌트 / 격자힌트).
  격자힌트 = "20×N 격자로 위치 추정 + 행/열 정렬 교차검증 + 컨테이너 포함관계 검증" 프롬프트
- **계측**: `eval/probe_recognition.py` (일회용, stdlib) — 텍스트 우선 greedy 매칭 후
  recall / centroid 오차(‰) / IoU / 엣지 recall
- 노드 규약: 화살표가 연결되는 흐름 참여자 = 노드(테두리 무관), 수식·삽입 figure = placeholder,
  순수 캡션 = annotation

## 4. 결과 (골든 v1.1 기준)

| 계측 | 박스 recall | 엣지 recall | centroid med/p90 (‰) | IoU med / <0.5 |
|---|---|---|---|---|
| ex01 무힌트 | **40/40 (100%)** | **22/22 (100%)** | 2.8 / 6.8 | 0.86 / 1 |
| ex01 격자힌트 | **40/40 (100%)** | **22/22 (100%)** | 2.2 / 6.3 | 0.89 / 1 |
| ex02 무힌트 | **15/15 (100%)** | **9/9 (100%)** | 3.0 / 7.0 | 0.92 / 0 |
| ex02 격자힌트 | **15/15 (100%)** | 8/9 (89%) | 3.2 / 11.5 | 0.88 / 1 |

- 계획의 공포 시나리오("좌표가 클러스터링의 행/열을 오분류시킬 만큼 노이즈")와 정반대 —
  **오차 중앙값이 장축의 0.2~0.3%**. 초기 임계 후보(X=15%)의 1/50 수준이다.
- 한글 텍스트(ex02: "Loss 계산", "θ 업데이트", "C번 반복" 등) 판독 문제 없음. 엣지 라벨·방향·
  순환 루프까지 정확.
- 인식측 잉여: ex02에서 캡션("CEM", "MPC", 파란 주석)을 노드로 잡는 해석 차이 2~7개 —
  오류가 아니라 규약 문제. M0b 인식 프롬프트에 "캡션 vs 흐름 참여자" 규약 명문화 필요.

## 5. 뜻밖의 주요 사건 — 골든 라벨 오류 4+1건 (R11의 실물 사례)

1차 계측에서 ex02 p90이 39.8‰로 튀었는데, 원인 추적 결과 **인식이 아니라 내 골든이 틀린 것**이었다:

| 항목 | 내 골든 (v1.0) | 진실 (픽셀 실측) | 발견 경로 |
|---|---|---|---|
| ex02 g3 (내부 프레임) | w=586 | **w=691** | 인식 2벌이 서로 일치·골든과만 상충 + 골든의 자식 포함관계 자기모순 |
| ex02 n12 (출력 수식) | x=843, w=117 | **x=869, w=81** | "⇒" 기호를 bbox에 오포함 |
| ex02 n4 (θ 업데이트) | x=268 | **x=257** | 인식 2벌 일치 |
| ex02 e8 (루프백) | n10→n8 | **n11→n8** | 인식 2벌 일치 — "엣지 누락 2건"으로 보였던 것의 실체 |
| ex01 n15 (Fused Feature) | [893,155] | **[882,153]** | 인식 2벌 일치, y는 골든이 맞았음(제3값) |

- 검증 절차: 인식 소스 간 교차중재로 의심 항목 도출 → **독립 검증 에이전트가 색상 마스크
  기반 픽셀 실측**으로 최종 판정 (자기승인 금지) → 골든 v1.1 정정 → 4벌 재계측.
- 교훈 (M0b로 전달): ① 골든 라벨도 확률적이다 — 계획 R11("라벨 품질이 게이트 신뢰도를
  좌우")이 반나절 만에 실증됨. ② **컨테이너⊇자식 포함관계 기계 검증**은 라벨 자기모순을
  공짜로 잡는다 → M0b 골든 프로토콜(golden-ir-labeling.md)에 필수 절차로 넣을 것.
  ③ 복수 독립 소스 교차중재가 편차 측정보다 강력한 라벨 QC였다.

## 6. 격자힌트 효과 — 채택 보류

ex01에서는 소폭 개선(2.8→2.2‰)이나 ex02에서는 오히려 악화(엣지 1개 누락, p90 상승).
표본 2장으로는 "노이즈 이내" — **M0b는 무힌트를 기본**으로 하고, 힌트는 M0b 표본 확대 시 재평가.
(계획 §7 2a의 "힌트 before/after 비교" 완료 — 승격 불필요 판정, API vision 승격도 불필요.)

## 7. 재현성 스팟체크 상태 (정직 기록)

계획 0a의 "재현성 스팟체크(필수)"용 독립 재라벨 에이전트는 장시간 무산출로 **사용자가 중지**
(취소 처리). 대체 증거로 라벨 신뢰도 상한은 확보됨: ① 독립 인식 2벌과 골든의 편차 중앙값
2.2~2.8‰ = (라벨 오차 + 인식 오차) 합의 상한, ② 분쟁 5건 픽셀 실측 판정 완료.
**정식 재라벨 편차 측정은 M0b step 0(전체 골든 라벨링)에 이월** — 그 시점에 어차피 골든을
확장 라벨링하므로 자연 흡수된다.

## 8. 판정과 다음 단계

- **M0a = GO.** 좌표 충실도 가망 없음(손절 조건) 해당 없음 — 오히려 인식은 이 파이프라인에서
  가장 약한 고리가 아닐 가능성이 높아졌다. 리스크 무게중심은 계획대로 R1(정규화 미감)·R13
  (커넥터 라우팅)으로 이동.
- **다음 = M0b** (계획 §4, 2~2.5일): 전체 골든(intent_align/size_class/role) + recognition_eval.py
  7지표 + 합성 노이즈 스윕 + 정규화 프로토타입(B1/B2 플래그) + 스텁 elbow 렌더러 + R13
  마이크로 스파이크(PowerPoint 저장 경로 생존 실측) + **존재적 AC2 게이트(재배치율, 최악치 집계)**.
  M0b의 AC2 판정과 R13 스파이크는 PowerPoint 실기 확인이 필요해 사용자 참여 지점이 있다.
- M0b 인식 프롬프트에 반영할 것: 캡션/흐름참여자 규약, 무힌트 기본.
